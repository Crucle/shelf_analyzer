# Анализатор прилавков

Программа проверяет фото прилавка: находит на нём отдельные товары,
распознаёт бренд с помощью мультимодальной модели ИИ и определяет,
стоят ли одинаковые товары единым блоком (правильная выкладка) или
разбросаны по прилавку (ошибка мерчендайзера).

## Как это устроено

```
фото
  │
  ▼
detector.py           — находит рамки отдельных товаров (проекция границ)
  │
  ▼
brand_classifier.py   — распознаёт бренд каждого товара (мультимодальная
  │                      модель CLIP: сравнивает фото с текстовым описанием
  │                      бренда, обучение под конкретные бренды не нужно)
  ▼
layout_checker.py     — проверяет, стоит ли каждый бренд единым блоком
  │
  ▼
visualizer.py + main.py — отчёт (JSON) + фото с разметкой
```

**Почему это мультимодальная модель**: CLIP обучен сопоставлять
изображения и текст в одном общем векторном пространстве. Чтобы узнать
бренд на фото, программа не обучается отдельно на фотографиях именно
этих брендов — она просто описывает каждый бренд текстовой фразой
("a photo of a bottle of Coca-Cola") и находит, какое описание ближе
всего к конкретной картинке товара (zero-shot классификация). Одна
модель одновременно "понимает" и картинку, и текст — в этом и есть
мультимодальность.

Детекция товаров (`detector.py`) осталась классической, без обучения —
её задача проще (найти сами товары, а не понять, что на них написано),
и с ней хорошо справляется анализ проекции границ (см. подробности ниже
в разделе "Реальные фото").

**История разработки**: изначально вместо CLIP использовалась
кластеризация без обучения по цвету/форме (`clustering.py` +
`features.py` — они остались в проекте, рабочие, просто больше не
задействованы в основном пайплайне `main.py`). Это было заменено на
мультимодальную модель, так как по заданию практики требовалось именно
её применение — эволюция подхода описана в отчёте по практике.

Каждый модуль решает свою задачу и не знает о деталях реализации
соседних — это сделано специально, чтобы части можно было заменять
независимо.

## Запуск

```bash
pip install -r requirements.txt
python main.py путь_к_фото.jpg --out result.jpg --brands "Coca-Cola,Sprite,Fanta,Schweppes,BonAqua"
```

При первом запуске CLIP скачает веса модели из интернета (~600 МБ
вместе с зависимостями torch/transformers) — нужен интернет один раз,
дальше веса кешируются на диске и работают офлайн.

Аргументы:
- `image` — путь к фото прилавка (обязательный)
- `--out` — куда сохранить фото с разметкой (по умолчанию `result.jpg`)
- `--brands` — список брендов через запятую, которые нужно искать на
  фото. Если не указать — используется список по умолчанию (напитки из
  тестового фото) **с встроенными подробными описаниями** каждого
  бренда (см. ниже) — они точнее простого списка названий. Если
  передать свой список через `--brands`, для каждого бренда будет
  использован общий шаблон фразы "a photo of a bottle of {бренд}" —
  этого достаточно для явно разноцветных товаров, но не для похожих
  друг на друга (см. ниже).
- `--crop x1,y1,x2,y2` — обрезать фото перед анализом до нужной области
  в пикселях. Если не указать, программа пробует обрезать фон
  автоматически (см. раздел "Обрезка фона" ниже).
- `--no-auto-crop` — отключить автоматическую обрезку фона (актуально,
  только если `--crop` не указан и автоматика ошибается на вашем фото).
- `--rows N` — сколько физических полок на фото (см. раздел ниже) —
  единственный параметр, который пока нельзя определить надёжно
  автоматически, нужно указывать вручную.
- `--references PATH` — папка с эталонными фото-примерами брендов (см.
  раздел про few-shot ниже).

### Точность распознавания похожих товаров

CLIP уверенно отличает товары с явно разным цветом (Coca-Cola, Sprite,
Fanta) даже по одной обычной фразе. А вот визуально похожие товары —
например, две прозрачные бутылки разных марок воды — по фразе вида
"a photo of a bottle of BonAqua" отличаются гораздо хуже: модели не
хватает сигнала, если единственное различие — мелкий текст на этикетке.

Решение — описывать не название бренда, а то, как он **выглядит**
(цвет крышки, цвет этикетки, форма логотипа), и можно указать сразу
несколько описаний на один бренд — они усредняются в один эталонный
вектор ("prompt ensembling", реализовано в `brand_classifier.py`). Так
устроен встроенный список по умолчанию в `main.py`:

```python
DEFAULT_BRANDS = {
    "Schweppes": [
        "a photo of a pale golden tonic water bottle with a dark rectangular label and yellow bottle cap",
        "a photo of a Schweppes bottle",
    ],
    "BonAqua": [
        "a photo of a clear still mineral water bottle with a blue label and blue bottle cap",
        "a photo of a BonAqua water bottle",
    ],
    # ...
}
```

Чтобы улучшить распознавание своих товаров, отредактируйте
`DEFAULT_BRANDS` в `main.py` (или передайте такой же словарь в
`analyze_image()`/`classify_brands()` напрямую, если пользуетесь кодом
как библиотекой) — опишите словами, чем ваш товар визуально отличается
от остальных в списке. В веб-версии (`app.py`) это тоже работает: если
не трогать поле со списком брендов, используются встроенные подробные
описания; если вписать свои названия — для них будет использован общий
шаблон фразы.

### Эталонные фото-примеры (few-shot) — если текстового описания мало

Для по-настоящему похожих товаров (например, Schweppes и BonAqua — обе
прозрачные бутылки, разница только в мелком тексте на этикетке) одних
текстовых описаний может не хватить. Более сильный вариант — показать
модели несколько реальных фото каждого бренда.

**Важно понимать, что это не дообучение весов модели** — веса CLIP не
меняются, не нужен ни градиентный спуск, ни большой датасет, ни GPU.
Технически эталонные фото прогоняются через ту же модель CLIP и
превращаются в такие же векторы, что и текстовые описания, — а дальше
они просто участвуют в том же усреднении ("prompt ensembling"), только
вперемешку с текстом. Это называется few-shot распознавание — быстрый,
лёгкий способ подсказать модели, ориентируясь на несколько примеров,
без полноценного обучения.

**В CLI**: положите несколько фото каждого проблемного бренда в папку
вида:
```
examples/
  Schweppes/
    фото1.jpg
    фото2.jpg
  BonAqua/
    фото1.jpg
```
и укажите папку через `--references`:
```bash
python main.py напитки.jpg --references ./examples --crop 460,0,1015,768 --rows 3
```
Название подпапки должно точно совпадать с названием бренда в списке
(`--brands` или значения по умолчанию). Для брендов без своей подпапки
просто не будет фото-эталонов — останутся только текстовые описания,
ничего не сломается.

**В веб-версии**: в боковой панели есть поле "Эталонные фото-примеры" —
загрузите туда несколько файлов с именами вида `Schweppes_1.jpg`,
`Schweppes_2.jpg`, `BonAqua_1.jpg` (часть имени до первого подчёркивания
должна совпадать с названием бренда).

Двух-трёх фото на бренд обычно достаточно, чтобы заметно поднять
точность на похожих товарах — но чем больше и разнообразнее примеры
(разный ракурс, освещение), тем устойчивее результат.

### Товары стоят вплотную без промежутка

Реальные фото с полностью заполненной полкой часто выглядят так: товары
одного бренда стоят впритык друг к другу, без видимого зазора. Обычный
метод детекции (по промежуткам между товарами, см. раздел "Реальные
фото" ниже) в этом случае видит только ОДИН большой блок на весь ряд
одинаковых товаров — если бы на этом не останавливались, все 8 бутылок
Coca-Cola подряд засчитались бы как один товар.

Решение в два шага:
1. **Разбиение по цвету** (`detector.py` → `_split_by_color`): между
   РАЗНЫМИ брендами, стоящими вплотную, почти всегда виден заметный
   переход цвета (красная Coca-Cola → зелёный Sprite), даже если сами
   бутылки физически соприкасаются. По этому переходу большой блок
   делится на под-блоки — по одному на бренд.
2. **Оценка количества по ширине** (`detector.py` → `_estimate_count`):
   внутри уже выделенного одноцветного под-блока (например, 8 бутылок
   Coca-Cola подряд) отдельные бутылки всё равно не разделяются
   поштучно — вместо этого их количество ОЦЕНИВАЕТСЯ делением ширины
   блока на типичную ширину одной бутылки (считается из высоты блока и
   усреднённого соотношения сторон бутылки).

**Это принципиально приблизительная оценка, а не точный поштучный
подсчёт** — и это специально видно в отчёте: поле `count_is_estimated`
становится `true`, в резюме появляется пометка "(оценочно)", а на
размеченном фото такие блоки подписываются как "Coca-Cola ×8" вместо
рамки на каждую бутылку отдельно. На тестовых фото с плотной выкладкой
оценка обычно отличается от реального количества на 10-25% — этого
достаточно, чтобы иметь ориентир, но не стоит выдавать это за точный
подсчёт в презентации или отчёте.

Программа печатает в консоль отчёт в формате дашборда (по образцу
референса, на который ориентировались при разработке):

```
============================================================
ОТЧЁТ О ВЫКЛАДКЕ
============================================================
Соответствие выкладке:   67%
Критических нарушений:   1
Полок распознано:        1
Товаров найдено:         6

На фото распознано полок: 1. Всего найдено товаров: 6, представлены
бренды: Coca-Cola, Fanta, Sprite. Соответствие выкладки оценено в 67% —
есть отдельные нарушения выкладки. Критических нарушений: 1.

КРИТИЧЕСКИЕ НАРУШЕНИЯ:
  — Товар "Coca-Cola" в строке 0 разбит на несколько блоков вместо одного.
```

а также JSON того же отчёта (для интеграции с другими системами или
приложения к отчёту по практике):

```json
{
  "compliance_percent": 67,
  "critical_violations_count": 1,
  "shelves_recognized": 1,
  "total_products": 6,
  "positions_detected": 6,
  "count_is_estimated": false,
  "brands_found": ["Coca-Cola", "Fanta", "Sprite"],
  "summary": "На фото распознано полок: 1. Всего найдено товаров: 6, ...",
  "critical_violations": ["Товар \"Coca-Cola\" в строке 0 разбит на несколько блоков вместо одного."],
  "warnings": []
}
```

**"Соответствие выкладке"** — своя метрика (см. `report.py`), а не
сравнение с настоящей планограммой (эталона "что где должно стоять" у
нас нет): это доля брендов, у которых не нашлось ни одного нарушения,
от всех найденных брендов. 100% — все бренды разложены единым блоком,
0% — у каждого найденного бренда есть хотя бы один разрыв.

**`total_products` vs `positions_detected`**: `positions_detected` —
это число найденных рамок (визуальных групп) на фото. `total_products`
— это сумма оценённого количества товаров во всех рамках; обычно равно
`positions_detected` (одна рамка = один товар), но может быть больше,
если товары стоят вплотную без зазора (см. раздел "Товары стоят
вплотную" выше) — тогда `count_is_estimated` становится `true`, и
`total_products` — это приблизительная оценка, а не точный подсчёт.

**`warnings`** — отдельно от нарушений выкладки: сюда попадает
предупреждение, если детектор находит подозрительно огромную (в
пересчёте на одну единицу товара) рамку — вероятно, в кадр попал лишний
фон, и стоит подрезать фото через `--crop`; либо предупреждение о том,
что часть количества в отчёте — приблизительная оценка (см. выше).

Программа также сохраняет фото, где рамки товаров подписаны названием
бренда и покрашены по нему же, а нарушения дополнительно обведены
красным.

## Весь стеллаж целиком и подсчёт товаров

Программа теперь умеет обрабатывать фото всего стеллажа за один запуск —
укажите количество физических полок через `--rows`:

```bash
python main.py напитки.jpg --out result.jpg --crop 460,0,1015,768 --rows 3 --brands "Coca-Cola,Sprite,Fanta,Schweppes,BonAqua"
```

Кадр (после `--crop`) делится на `--rows` равных горизонтальных полос, и
детекция запускается на каждой полосе отдельно (см. "Реальные фото" ниже
про то, почему это нужно), а результаты объединяются в одну общую
разметку и один отчёт.

Программа явно печатает общее количество найденных товаров и
распознанные бренды:

```
Найдено товаров на полке: 16
Распознанные бренды: BonAqua, Coca-Cola, Fanta, Schweppes, Sprite
```

Ценники и подписи полок в это число не входят — они отфильтровываются
на этапе детекции (удаление длинных линий + фильтр по пропорциям), это
проверено на тестовых фото.

Если полки на фото расположены неравномерно (`--rows` даёт не совсем
точные границы), встречаются два нередких артефакта:
- товар может слипнуться с соседним в один бокс, если между ними мало
  видимого просвета на конкретном фото;
- если в кадр попал самый край металлической рамки полки, он иногда
  засчитывается как отдельный мелкий "товар". Обрежьте `--crop` чуть уже
  по ширине, чтобы полностью убрать рамку из кадра.

## Сравнение с планограммой (отсутствующие, лишние, порядок)

Кроме проверки "одинаковые товары стоят единым блоком" (см. выше),
программа умеет сравнивать фактическую выкладку с **планограммой** —
эталонной схемой, где для каждой полки заранее задано, какие бренды и
в каком порядке должны на ней стоять. Реализовано в `planogram.py`.

Сравнение находит три независимых вида нарушений на каждой полке:
- **отсутствующие товары** — есть в планограмме, но не найдены на фото
  (потенциальный признак того, что товар закончился на складе);
- **лишние товары** — найдены на фото, но их не должно быть по
  планограмме;
- **неправильный порядок** — все нужные товары на полке есть (или
  почти все), но расположены не в том порядке, что предусмотрено
  планограммой.

Формат планограммы — обычный JSON: список списков названий брендов,
один список на полку, сверху вниз, товары слева направо:

```json
[
  ["Coca-Cola", "Coca-Cola", "Coca-Cola", "Sprite", "Sprite", "Fanta"],
  ["Schweppes", "Schweppes", "BonAqua", "BonAqua"]
]
```

**В CLI** — сохраните такой файл (например, `planogram.json`) и
передайте через `--planogram`:

```bash
python main.py напитки2.jpg --out result.jpg --rows 2 --planogram planogram.json
```

**В веб-версии** — в боковой панели есть блок "Планограмма": можно
либо загрузить готовый JSON-файл, либо вписать ожидаемые бренды по
каждой полке вручную прямо в текстовые поля (через запятую) и потом
скачать получившуюся планограмму как JSON-файл, чтобы переиспользовать
её для следующих фото того же прилавка — не нужно вписывать заново
каждый раз.

Если планограмму не указывать (ни в CLI, ни в веб-версии) — эта
проверка просто не выполняется, а остальной анализ (детекция,
распознавание бренда, проверка на разбитые блоки, отчёт) работает как
обычно, без изменений.

## Веб-версия (Streamlit)

Кроме консольной `main.py`, есть веб-интерфейс `app.py`: загрузка фото
через браузер, ползунки настроек (обрезка, число полок, список брендов,
порог уверенности модели) и наглядный результат — без единой команды в
терминале.

### Запуск локально

```bash
pip install -r requirements.txt
streamlit run app.py
```

Откроется браузер с адресом вида `http://localhost:8501`. При первом
клике "Анализировать" модель CLIP скачается из интернета — это может
занять минуту-две, дальше она закеширована и грузится мгновенно.

### Публикация на Streamlit Community Cloud (бесплатно)

1. **Создайте репозиторий на GitHub** и загрузите в него всю папку
   `shelf_analyzer` целиком (все `.py` файлы + `requirements.txt`).
   Проще всего через веб-интерфейс GitHub: "Add file" → "Upload files" →
   перетащить все файлы → Commit. Либо через git:
   ```bash
   cd shelf_analyzer
   git init
   git add .
   git commit -m "Анализатор прилавков"
   git branch -M main
   git remote add origin https://github.com/ВАШ_ЛОГИН/shelf-analyzer.git
   git push -u origin main
   ```
2. Зайдите на **[share.streamlit.io](https://share.streamlit.io)** и
   войдите через свой аккаунт GitHub.
3. Нажмите **"New app"**.
4. Выберите: ваш репозиторий → ветку `main` → путь к файлу `app.py`.
5. Нажмите **"Deploy"**. Первый запуск займёт несколько минут — ставятся
   зависимости из `requirements.txt`, включая torch и transformers (они
   заметно тяжелее, чем opencv/numpy/scikit-learn, поэтому сборка
   медленнее, чем была раньше).
6. После сборки приложение получит постоянную ссылку вида
   `https://ВАШ-АРР-ИМЯ.streamlit.app` — её можно вставить в отчёт по
   практике или отправить любому человеку без установки Python.

**Важно про размер зависимостей:** `torch` + `transformers` — тяжёлые
библиотеки (сотни МБ), а бесплатный тариф Streamlit Community Cloud
ограничен по памяти. Если увидите ошибку нехватки ресурсов при деплое —
самое надёжное решение: указать CPU-версию torch явно, добавив в
`requirements.txt` вместо строки `torch>=2.0` две строки:
```
--extra-index-url https://download.pytorch.org/whl/cpu
torch>=2.0
```
Это ставит облегчённую сборку torch без ненужной здесь поддержки GPU.

**Важно:** в `requirements.txt` используется `opencv-python-headless`,
а не `opencv-python` — обычная версия тянет системные GUI-библиотеки
(`libGL`), которых нет в облачном окружении Streamlit Cloud, и деплой
падает с ошибкой `ImportError: libGL.so.1`. Headless-версия работает
идентично для всех операций, которые использует этот проект (нет ни
одного вызова `cv2.imshow` или похожих GUI-функций).

Если после деплоя приложение всё равно не запускается — откройте логи
через "Manage app" в правом нижнем углу интерфейса Streamlit Cloud, там
обычно точно видно, какого пакета не хватает.


## Реальные фото: важные особенности

**Алгоритм детекции полностью переработан** (вместо поиска замкнутых
контуров — анализ проекции границ по столбцам и строкам). Новый подход
гораздо надёжнее: на тестовых фото полок он находит 5/5, 6/6 товаров
на полку, включая товары со сложной, "плотной" этикеткой (Coca-Cola) и
прозрачные бутылки — раньше это были главные слабые места.

**Всё ещё лучше всего работает на фото ОДНОЙ полки**, а не всего стеллажа
целиком — принцип работы (проекция границ по всей высоте кадра) требует,
чтобы в кадре была только одна "линия" товаров. Через `--crop` вырежьте
одну полку за раз:

```bash
python main.py напитки.jpg --out result.jpg --crop 460,0,1015,260
```

(числа `x1,y1,x2,y2` подбираются под конкретное фото).

Оставшиеся редкие огрехи:
- бутылки с выраженной "талией" (например, Sprite) иногда режутся на два
  фрагмента в самом узком месте — это можно смягчить, увеличив аргумент
  `gap_merge` для строк внутри `detector.py`, если это станет мешать;
- если в кадр попадает край металлической рамки стеллажа или текст
  подписи полки прямо по границе кропа — он может дать один лишний
  мелкий бокс. Обрежьте кроп чуть теснее, чтобы убрать такие края.

## Обрезка фона (автоматически или вручную)

**Теперь это происходит автоматически.** Если не указать `--crop`
вручную, программа сама пробует найти сам прилавок и обрезать
окружающий фон (соседние стеллажи, проход между рядами) — см.
`detector.py` → `auto_crop`.

Работает на простом наблюдении: на большинстве товарных фото сам
прилавок снят в фокусе, а окружение вокруг намеренно размыто (боке),
чтобы не отвлекать внимание. Резкость (дисперсия оператора Лапласа)
заметно выше в зоне фокуса, чем в размытом фоне — по этой разнице и
находится граница.

```bash
python main.py напитки.jpg --out result.jpg --rows 3
# --crop не указан — программа сама обрежет фон
```

Если фото снято без боке (всё в фокусе, например обычное фото с
телефона мерчендайзера) — авто-обрезка не найдёт чёткой границы и
честно не станет резать наугад, просто возьмёт всё фото целиком. В этом
случае (или если автоматика ошиблась) обрежьте вручную:

```bash
python main.py напитки.jpg --out result.jpg --crop 440,0,1040,768
```

Числа `x1,y1,x2,y2` — координаты левого верхнего и правого нижнего угла
нужной области в пикселях. Проще всего подобрать их, открыв фото в
любом просмотрщике с линейкой/координатами (например, Paint показывает
координаты курсора внизу). Чтобы полностью отключить автообрезку (даже
если `--crop` не указан) — используйте `--no-auto-crop`.

В веб-версии (`app.py`) автообрезка тоже включена по умолчанию:
ползунки обрезки сразу выставляются в подобранное автоматически
положение при загрузке фото, а кнопка "Сбросить на всё фото" возвращает
их к исходным границам, если нужно поправить вручную.

Также детектор слабо ловит **прозрачные и сильно бликующие** предметы
(бутылки с водой, стекло) — у них мало чётких границ для Canny. Это
тоже ограничение подхода без обучения, см. пункт про YOLO ниже.

## Текущие ограничения

- **Детекция** работает через анализ проекции границ (см. `detector.py`).
  Хорошо подходит для одной полки за раз; на плотных, неровных или
  сильно бликующих полках потребуется подстройка параметров
  (`min_area_ratio`, `max_area_ratio`, пороги Canny, `gap_merge`).
- **Распознавание бренда** — CLIP работает по принципу "выбери наиболее
  похожий бренд из заданного списка" (zero-shot). Если на фото товар,
  которого в списке `--brands` нет, модель всё равно навяжет ему
  какой-то бренд из списка (частично смягчается порогом
  `min_confidence` — см. `brand_classifier.py`). Чем ближе список
  брендов к реальному ассортименту на фото, тем точнее результат.
- **Проверка выкладки** сейчас построчная: строки полки определяются по
  Y-координате центра товара, и правило "один бренд — один непрерывный
  блок в строке" проверяется отдельно для каждой строки. Проверка
  группировки *между* рядами (например, один и тот же бренд должен
  стоять в одной и той же колонке на всех полках) пока не реализована —
  это следующий логичный шаг.

## Куда развивать дальше

1. **Точность распознавания брендов**: дообучить CLIP (fine-tuning) на
   реальных фото конкретных брендов, если такой датасет появится — это
   должно заметно поднять точность по сравнению с zero-shot режимом,
   особенно для визуально похожих товаров одного бренда (обычная и zero
   версии, разные объёмы).
2. **Детекция**: заменить `detect_products()` в `detector.py` на вызов
   обученной модели (например, YOLOv8, дообученной на фото прилавков) —
   если проекционный метод перестанет справляться на более сложных фото.
   Сигнатура функции (фото → список рамок) остаётся прежней, остальной
   код трогать не нужно.
3. **Автоматическое разбиение на полки**: сейчас число полок (`--rows`)
   нужно указывать вручную, автоматическое разбиение по фото оказалось
   ненадёжным (см. историю разработки) — можно вернуться к этой задаче
   отдельно, например через детекцию самих полочных стоек/направляющих.
4. **Отчёт**: сейчас "соответствие выкладке" — собственная метрика (см.
   `report.py`, `_compliance_percent`). Если появится настоящая
   планограмма (эталон "что где должно стоять"), эту функцию легко
   заменить на честное сравнение факт/план — набор полей отчёта
   (`compliance_percent`, `critical_violations` и т.д.) можно оставить
   тем же, чтобы не переделывать `app.py` и вывод в `main.py`.
